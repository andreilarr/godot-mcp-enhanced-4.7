## 1. 概述与架构

godot-mcp-enhanced 是增强版 Godot MCP 服务器，提供 **140+ 工具** 覆盖 AI 辅助游戏开发的完整闭环：理解项目 → 编辑场景/脚本 → 执行验证 → 运行时调试。

### 双模式架构

服务器同时支持两种连接 Godot 的方式，**自动检测、互不冲突**：

| 模式 | 连接方式 | 适用场景 |
|------|----------|----------|
| **Headless CLI**（默认） | spawn 一个 headless Godot 子进程执行 GDScript | 离线代码执行、场景编辑、语法验证、批量操作 |
| **Editor WebSocket** | JSON-RPC 2.0 连接编辑器内插件 | 实时操作开发者**当前打开**的场景（所见即所得） |

大多数工作流用 Headless 模式即可。Editor 模式适合"在编辑器里实时调整正在编辑的场景"。

### 工具组织（v0.18.0 起）

为降低 MCP 客户端的工具发现成本，v0.18.0 将 **39 个独立工具合并为 27 个工具组**（按 `action` 参数分发）。例如：

- `scene(action="read_scene" | "create_scene" | "add_node" | ...)`
- `script(action="read_script" | "write_script" | "edit_script" | ...)`
- `runtime(action="run_project" | "get_debug_output" | "record_*" | ...)`
- `validation(action="run_and_verify" | "validate_scripts" | "verify_delivery" | ...)`
- `game(action="game_query" | "game_input" | "game_wait" | ...)`

> 旧工具名仍可用（设 `GODOT_MCP_WARN_LEGACY=1` 打警告），调用 `manage_tools(action="migrate")` 获取完整迁移映射。

## 2. 安装与配置

### 2.1 Claude Code 全局安装（推荐）

```bash
claude mcp add -s user godot -- npx -y godot-mcp-enhanced
```

`-s user` 写入 `~/.claude.json` 顶层，**所有 Godot 项目自动可用**。

### 2.2 Godot 二进制查找优先级

```
工具参数 godot_path  >  项目配置文件  >  GODOT_PATH 环境变量  >  PATH  >  平台搜索
```

**多版本 Godot**：在项目目录创建 `.godot/mcp-godot.json`：
```json
{ "version": 1, "godot_path": "/path/to/Godot_v4.6.3-stable_macos.arm64" }
```
或在 `project.godot` 末尾加：
```ini
[godot_mcp]
godot_path=/path/to/godot
```
支持 [godots](https://github.com/MakovWait/Godots) 版本管理器（项目根 `.godot-version` 文件）。

### 2.3 首次使用：一键配置项目规则

连接项目后立即运行：
```
project(action="setup_project_rules", project_path="你的项目路径")
```
自动生成：
- `.claude/settings.json`：PostToolUse hook，编辑 `.gd` 后自动提醒 AI 验证语法
- `CLAUDE.md`：GDScript 验证规则 + 发版门禁

已有配置用 `force=true` 覆盖。

## 3. 核心概念（必读）

> 这一节是 AI 正确使用工具的关键。多数"工具不工作"的困惑源于对这些概念的误解。

### 3.1 `project_path` —— 几乎所有工具的第一个参数

每个工具都接受 `project_path`，指向 Godot 项目根目录（含 `project.godot`）。

**优先级**：工具参数 → `GODOT_PROJECT_PATH` 环境变量 → 从 cwd 向上搜索 `project.godot`（最多 30 层）。

⚠️ **建议总是显式传 `project_path`**。仅靠 cwd 自动检测在 IDE/MCP 客户端里不可靠（cwd 常是客户端启动目录而非项目目录）。

### 3.2 路径约定：`res://` vs 绝对路径

| 工具 | 路径参数 | 约定 |
|------|----------|------|
| `scene(action="read_scene")` | `scene_path` | **`res://` 相对路径**（如 `res://scenes/main.tscn`）|
| `script(action="read_script")` | `script_path` | **`res://` 相对路径** |
| `script(action="write_script")` | `script_path` | `res://` 或项目相对路径 |
| 其他大部分工具 | `project_path` | **绝对文件系统路径** |

⚠️ **陷阱**：`read_scene`/`read_script` 传绝对路径会报 `project_path is required`——它们要的是 `res://` 路径 + `project_path` 绝对路径。

### 3.3 Headless 执行 vs 持久化

这是最容易混淆的概念：

- **持久化操作**（改 .tscn/.gd 文件）：`create_scene`/`add_node`/`edit_node`/`save_scene`/`write_script`/`edit_script` —— **写入磁盘**，重启后保留。
- **运行时操作**（改场景树实例）：`signal_connect`/`physics_raycast`/`audio_play`/`tilemap_set_cell`/`material_write`/`particles_create`/`nav_*`/`animtree_*`/`ik_*` —— **只在当前 headless 执行上下文生效，不写盘**。进程结束即消失。

> 需要持久化的场景修改，用 `add_node` + `save_scene`，不要用运行时操作。

### 3.4 `execute_gdscript` 的代码片段模式

无需写 `extends`，输入的代码会被自动包装为 `extends SceneTree` 脚本：
- `func`/`var`/`const`/`class_name` 等声明 → 放在类级别
- 普通语句行 → 放进 `_initialize()` 体内执行
- 用 `_mcp_output(key, value)` 返回结构化结果

```
script(action="execute_gdscript", code="
var x: int = 6
var y: int = 7
_mcp_output('sum', str(x + y))
")
```

⚠️ **编译错误传染**：headless Godot 加载项目时会编译**整个项目**的所有脚本。如果项目里有任何脚本编译错误（哪怕与你的代码无关），`execute_gdscript` 会整体失败。先用 `validate_scripts` 排查项目脚本健康度。详见 [故障排查 §12.2](#122-execute_gdscript-报项目脚本的编译错误)。

### 3.5 确认令牌

危险/有副作用的操作（如 `execute_gdscript`、`remove_node`）首次调用返回 `confirmation_token`，需用 `confirm_and_execute(token=...)` 二次确认。这是防 AI 误操作的保护层。

## 4. 工作流一：理解现有项目

接到一个陌生 Godot 项目时，按此顺序建立理解：

### 4.1 项目元数据

```
project(action="get_project_info", project_path="D:/my-game")
```
返回：项目名、Godot 版本、autoload 列表、input 映射、渲染器配置、**文件统计**（多少 .gd/.tscn/.png）。

### 4.2 读取场景结构（v0.18.1 修复后正确解析属性）

```
scene(action="read_scene", project_path="D:/my-game", scene_path="res://scenes/main.tscn")
```

返回结构化节点树。**v0.18.1 起，节点属性被正确解析为类型化对象**：

```json
{
  "header": { "load_steps": 2, "format": 3 },
  "extResources": [{ "type": "Script", "path": "res://main.gd", "id": "1_main" }],
  "nodeTree": [{
    "name": "Main", "type": "Node2D", "parent": "",
    "properties": [
      { "name": "script", "value": { "__type": "ExtResource", "id": "1_main" } },
      { "name": "position", "value": { "__type": "Vector2", "value": "100, 200" } },
      { "name": "modulate", "value": { "__type": "Color", "value": "1, 0, 0, 1" } },
      { "name": "text", "value": "Hello" },
      { "name": "font_size", "value": 24 },
      { "name": "theme_override_font_sizes/font_size", "value": 24 }
    ],
    "children": [...]
  }],
  "connections": [{ "signal": "pressed", "from": "UI/Button", "to": ".", "method": "_on_button_pressed" }]
}
```

**支持的属性类型**：`ExtResource`、`SubResource`、`Color`、`Vector2`、`Vector3`、`NodePath`、数组、字典、整数、浮点、布尔、字符串。含 `/` 的属性名（如 `theme_override_styles/panel`）保留完整路径。

> 只需拓扑？传 `summary_only=true` 返回精简的节点名/类型树，不含属性。

### 4.3 读取脚本

```
script(action="read_script", project_path="D:/my-game", script_path="res://scripts/player.gd")
```
返回脚本内容 + 元数据（行数、类名、是否 autoload）。

### 4.4 列出文件

```
project(action="list_files", project_path="D:/my-game", extensions=[".gd"], subdirectory="scripts")
```
扩展名 + 子目录过滤。返回绝对路径数组。

### 4.5 解析项目配置

```
project(action="read_project_config", project_path="D:/my-game")
```
把 `project.godot` 解析为结构化 JSON（autoload/input/display/rendering 各段）。

## 5. 工作流二：创建与编辑场景

### 5.1 创建场景

```
scene(action="create_scene", project_path="D:/my-game",
      scene_path="res://scenes/player.tscn", root_type="CharacterBody2D", root_name="Player")
```

### 5.2 添加节点（单个）

```
scene(action="add_node", project_path="D:/my-game", scene_path="res://scenes/player.tscn",
      node_type="Sprite2D", node_name="Sprite", parent_node_path="Player",
      properties={ "position": [100, 200], "texture": "res://assets/player.png" })
```

`properties` 值经智能类型转换（`smart-coerce`）：`[100,200]` → `Vector2`、`"res://x.png"` → 资源加载、`[1,0,0,1]` → `Color`。

### 5.3 批量添加节点（性能推荐）

```
scene(action="batch_add_nodes", project_path="...", scene_path="...",
      nodes=[
        { "node_type": "Label", "node_name": "Title", "properties": {...} },
        { "node_type": "Button", "node_name": "StartBtn", "parent_node_path": "UI", "properties": {...} }
      ])
```

**只启停一次 headless Godot**，比循环调 `add_node` 快一个数量级。

### 5.4 编辑节点属性

```
scene(action="edit_node", project_path="...", node_path="Player/Sprite",
      properties={ "position": [200, 300], "modulate": [1, 1, 1, 0.5] })
```

### 5.5 保存

```
scene(action="save_scene", project_path="...", scene_path="res://scenes/player.tscn")
```
⚠️ 改动**不会自动保存**，必须显式调 `save_scene`。

### 5.6 实例化与分离

- `instance_scene`：把一个 .tscn 实例化到目标父节点
- `detach_instance`：把实例节点从场景树分离（转为独立节点）
- `merge_scene`：三方合并 .tscn（自动重映射 ExtResource/SubResource ID）

## 6. 工作流三：编写与编辑脚本

### 6.1 写入新脚本

```
script(action="write_script", project_path="D:/my-game",
      script_path="res://scripts/player.gd",
      content="extends CharacterBody2D\n\n@export var speed: float = 200.0\n\nfunc _physics_process(delta):\n\tvelocity = Vector2.ZERO\n\tmove_and_slide()\n")
```
`overwrite=false`（默认）时已存在文件会拒绝，防意外覆盖。

### 6.2 按行范围编辑

```
script(action="edit_script", project_path="...", script_path="res://scripts/player.gd",
      start_line=10, end_line=15, new_content="\t# 新逻辑\n\tspeed = 300.0\n", indent_mode="smart")
```
`indent_mode`: `raw`（原样）或 `smart`（按上下文自动缩进）。`verify_content` 守卫可验证期望内容、`auto_validate` 自动跑语法检查并在失败时回滚。

### 6.3 搜索替换（CRLF 安全）

```
script(action="edit_script", project_path="...", script_path="...",
      search_and_replace={ "search": "speed = 200.0", "replace": "speed = 300.0", "occurrence": 0 })
```
`occurrence`: `0` = 全部，`1` = 第 1 处。CRLF 行尾自动归一化匹配。

### 6.4 全项目批量替换

```
script(action="project_replace", project_path="D:/my-game",
      search="get_node(\"Player\")", replace="$Player",
      extensions=[".gd"], dry_run=true)
```
先 `dry_run=true` 预览，确认后 `dry_run=false` 写入。原子写 + 失败回滚。

### 6.5 生成测试

```
script(action="generate_test", project_path="...", script_path="res://scripts/player.gd")
```
分析脚本生成 GUT 测试骨架。

## 7. 工作流四：执行与验证

### 7.1 动态执行 GDScript

见 [§3.4](#34-execute_gdscript-的代码片段模式)。设置 `load_autoloads=true` 可在完整 Autoload 上下文运行（访问 DataRegistry/PlayerData 等单例，但更慢）。

### 7.2 一键运行验证（最常用）

```
validation(action="run_and_verify", project_path="D:/my-game", capture_tree=true)
```
headless 启动项目 → 捕获输出 → 结构化错误分析。自动检测：
- 脚本编译错误
- 资源缺失
- 版本不一致（project.godot vs 二进制）

`capture_tree=true` 同时返回运行时场景树快照。

### 7.3 单文件语法验证

```
validation(action="validate_scripts", project_path="D:/my-game",
           scripts=["scripts/player.gd", "scripts/enemy.gd"])
```
对每个脚本执行 Godot `load()` 编译检查（触发完整编译，含跨文件依赖解析），能发现 headless 运行可能遗漏的 Parse Error。不传 `scripts` 则扫全部。

### 7.4 错误分析

```
validation(action="analyze_error", output="<粘贴 Godot 输出文本>")
```
重新分析任意 Godot 输出，提供结构化错误 + 修复建议。

### 7.5 项目完整性检查

```
project(action="validate_project", project_path="D:/my-game")
```
检查：.tscn 引用了不存在的资源、.gd 的 `preload()`/`load()` 路径无效、源资源删除后残留 .import 文件。

### 7.6 端到端交付验证

```
validation(action="verify_delivery", project_path="D:/my-game",
           scene="res://scenes/main.tscn", assertions=[
             { "gdscript": "assert(GameManager != null)", "expect": "true" }
           ])
```
四维度：场景树完整性 + 脚本健康 + 性能 + 自定义断言 + GDD 合规。

### 7.7 运行 GUT 单元测试

```
runtime(action="run_tests", project_path="D:/my-game", test_script="res://test/")
```
解析 GUT 测试结果为结构化 pass/fail。

## 8. 工作流五：运行时调试（Game Bridge）

Game Bridge 让 AI 能与**正在运行的游戏**交互：查询状态、发送输入、等待条件。需要先安装 Bridge autoload。

### 8.1 安装 Bridge

```
game(action="game_bridge_install", project_path="D:/my-game")
```
拷贝 `mcp_bridge.gd` 到项目 + 注册 autoload。它启动一个 **TCP 服务（127.0.0.1:9081）** 供 MCP server 连接。

### 8.2 查询运行时状态

```
game(action="game_query", method="get_tree", project_path="D:/my-game")
game(action="game_query", method="get_node_properties",
     params={ "path": "root/Player" }, project_path="D:/my-game")
game(action="game_query", method="take_screenshot",
     params={ "path": "user://debug.png" }, project_path="D:/my-game")
```
可用方法：`ping`、`get_tree`、`find_nodes`、`get_node_properties`、`get_performance`、`get_viewport_info`、`take_screenshot`。

### 8.3 发送输入

```
game(action="game_input", method="send_key", params={ "key": "ui_accept", "pressed": true })
game(action="game_input", method="send_mouse_click", params={ "x": 640, "y": 360, "button": "left", "pressed": true })
game(action="game_input", method="send_text", params={ "text": "hello" })
```

### 8.4 等待条件（v0.18.1 修复后真正轮询）

⚠️ **v0.18.0 及之前**：`game_wait` 是单次快照，条件不成立立即返回 false，命名"wait"但实际不等待，会破坏自动化。

**v0.18.1 起**：在 `timeout` 窗口内按 `interval_ms` 反复探测，直到条件成立或超时：

```
game(action="game_wait", method="wait_for_node",
     params={ "path": "root/BattleScene" },
     timeout=5000, interval_ms=200, project_path="D:/my-game")
```

返回值：
```json
// 条件成立
{ "exists": true, "wait_completed": true, "elapsed_ms": 412 }
// 超时
{ "exists": false, "wait_completed": false, "timed_out": true, "elapsed_ms": 5000 }
```

| 参数 | 说明 | 默认 |
|------|------|------|
| `timeout` | 轮询总预算（毫秒），上限 60000 | 10000 |
| `interval_ms` | 探测间隔（毫秒），范围 50–2000 | 200 |

`wait_for_property` 同理：在窗口内轮询 `str(current) == str(expected)`。

> **前提**：游戏必须正在运行且 Bridge autoload 已加载。手动启动游戏后才能连 Bridge。

## 9. 闭环开发完整示例

```
1. 理解项目
   project(action="get_project_info", ...)          → 元数据 + 文件统计
   scene(action="read_scene", scene_path="res://scenes/player.tscn")
                                                     → 节点树 + 属性（v0.18.1 正确解析）
   script(action="read_script", script_path="res://scripts/player.gd")
                                                     → 当前代码

2. 编辑
   script(action="edit_script", search_and_replace={...})  → 改代码
   scene(action="add_node", ...)                    → 加节点
   scene(action="save_scene", ...)                  → 持久化

3. 验证
   validation(action="validate_scripts", scripts=[...])   → 单文件语法
   validation(action="run_and_verify", capture_tree=true) → headless 运行 + 错误分析

4. 运行时确认（可选）
   runtime(action="run_project", ...)               → 启动游戏
   game(action="game_wait", method="wait_for_node", ...)  → 等待场景加载
   game(action="game_query", method="take_screenshot", ...) → 截图核对

5. 收尾
   project(action="validate_project", ...)          → 资源/引用完整性
```

## 10. v0.18.1 关键修复

本次发布修复了功能验证审查发现的 3 个阻塞性 CRITICAL：

| 修复 | 影响工具 | 变化 |
|------|----------|------|
| **parseTscn 属性解析** | `read_scene` | 节点多行属性（`position`/`script`/`color`/`texture` 等）现在解析为类型化对象，而非整行字符串。ExtResource/Color/Vector/NodePath/数组/字典全部正确识别。 |
| **parseTscn 头部解析** | `read_scene` | `header` 现在返回 `{format, load_steps, uid}`，不再是 `{}`。 |
| **game_wait 轮询** | `game_wait` | `wait_for_node`/`wait_for_property` 现在在 timeout 窗口内真正轮询，不再单次快照。新增 `interval_ms` 参数。 |

**迁移**：依赖 `game_wait` 的旧代码若假设"立即返回"——现在它会阻塞到条件成立或超时。若需旧行为（单次检查），设 `timeout=interval_ms`（如都设 200）。

详见 [CHANGELOG.md](../CHANGELOG.md)。

## 11. 安全模型

本工具设计为 **AI 与开发者在同一台受信任机器上协作**，不提供远程认证。理解安全边界：

### 11.1 路径访问控制（deny-by-default）

```
优先级：GODOT_MCP_UNRESTRICTED=true → 全放开（仅开发环境）
       > ALLOWED_PROJECT_PATHS=/path1;/path2 → 仅允许列出根及子目录
       > 未配置 → 限制到 process.cwd()（默认）
```

防护层：UNC 路径拒绝 → Windows 设备名拒绝（CON/PRN/AUX/NUL/COM*/LPT*）→ 迭代 URL 解码防 `%2e%2e` → `..` 段拒绝 → realpath + relative 兜底（防符号链接）。

实测：`read_script(script_path="res://../../../etc/passwd")` → `"Path traversal detected"`。

### 11.2 GDScript 沙箱（防误操作，非安全边界）

`execute_gdscript` 在执行前扫描危险模式：`OS.execute`/`FileAccess(WRITE)`/`DirAccess.remove`/`str2var`/`ClassDB`/`call("string")`/`Expression.execute` 等，并检测字符串拼接绕过（`"OS" + ".execute"`）。

⚠️ **已知局限**：GDScript 图灵完备，正则无法穷举绕过（变量间接、多步构造）。沙箱仅防 AI **意外**调用危险 API，**不能防御恶意构造**。

控制开关：
- `GODOT_MCP_SANDBOX=disabled` → 关闭扫描（⚠️ 完全放开，仅调试用）
- `GODOT_MCP_SANDBOX=strict` → 连 `FileAccess.open`（只读）也拦
- `GODOT_MCP_ALLOW_UNSAFE=true` → 允许被拦操作（仅开发环境）

### 11.3 输出标记防伪造

每次 `execute_gdscript` 用 `crypto.randomUUID()`（122 位熵）生成标记，GDScript 代码无法伪造 MCP 输出。

### 11.4 Game Bridge 密钥

Bridge autoload 生成随机密钥文件（`.godot/mcp_bridge_9081.secret`），**读后即删**，TCP 仅绑定 `127.0.0.1`。

### 11.5 多用户/不可信环境

如需真正的隔离，用 **容器/VM** + `GODOT_MCP_ALLOW_UNSAFE=false`，不要依赖沙箱。

## 12. 故障排查

### 12.1 `read_scene`/`read_script` 报 `project_path is required`

这两个工具要 **`res://` 相对路径** + **`project_path` 绝对路径**，不接受裸绝对路径的 scene_path/script_path。

❌ `read_scene(scene_path="D:/game/scenes/main.tscn")`
✅ `read_scene(project_path="D:/game", scene_path="res://scenes/main.tscn")`

### 12.2 `execute_gdscript` 报项目脚本的编译错误

现象：你执行的代码很简单（如 `var x = 1`），却返回一堆 `Identifier "XxxManager" not declared`、`Could not preload resource`。

根因：headless Godot 加载项目时编译**整个项目**所有脚本，任一脚本的错误都会让整体失败。

解决：
1. 先 `validation(action="validate_scripts")` 找出报错脚本
2. 修复项目脚本，或临时设 `load_autoloads=false`（不加载 autoload 上下文）
3. 重试 `execute_gdscript`

### 12.3 `game_wait` 立即返回 false / 不等待

- v0.18.0 及之前：这是已知 bug（单次快照），升级到 **v0.18.1+**。
- v0.18.1+ 仍立即返回：检查游戏是否**正在运行** + Bridge autoload 是否加载。Bridge 只连运行中的游戏。

### 12.4 `another Godot process is running`

长运行锁（`run_project`）被占用。先 `runtime(action="stop_project")` 释放，或等待自动死锁检测（>5min 进程死亡自动释放）。

短操作（`query_scene_tree`）并发上限 3，超过会排队。

### 12.5 工具改动未生效（重启 MCP server）

修改了 MCP server 源码或 `npm update` 后，**当前会话的 MCP server 进程仍是旧的**（启动时 spawn，不热重载）。需在客户端重连 MCP 或重启会话。

### 12.6 截图黑屏（Windows）

Windows headless 模式下 viewport 纹理返回 null。`capture_screenshot` 自动用窗口模式（GPU 上下文）截图。若仍黑屏，确保项目渲染器非纯 headless。

### 12.7 运行时操作不持久化

`signal_connect`/`tilemap_set_cell`/`particles_create` 等运行时操作只影响当前执行上下文。需要持久化：用 `add_node`/`write_script` + `save_scene`。

### 12.8 项目路径缓存（30 秒）

`resolveProjectScene` 有 30s TTL 缓存。切换项目后等 30s 或重启 MCP server。

## 13. 环境变量参考

| 变量 | 说明 | 默认 |
|------|------|------|
| `GODOT_PATH` | Godot 可执行文件路径 | 自动搜索 |
| `GODOT_PROJECT_PATH` | 默认项目路径 | cwd 向上搜索 |
| `GODOT_MCP_SEARCH_PATHS` | 额外 Godot 搜索目录（分号分隔） | — |
| `ALLOWED_PROJECT_PATHS` | 允许的项目根（分号分隔） | 未配=限制 cwd |
| `GODOT_MCP_UNRESTRICTED` | `true`=禁用路径限制（⚠️ 仅开发） | false |
| `GODOT_MCP_SANDBOX` | `disabled`/`strict`/默认 | 默认（扫描） |
| `GODOT_MCP_ALLOW_UNSAFE` | `true`=允许被拦危险操作（⚠️ 仅开发） | false |
| `GODOT_MCP_WARN_LEGACY` | `1`=旧工具名打 warning | 0 |
| `GODOT_MCP_MULTI_INSTANCE` | `true`=多实例模式（⚠️ HTTP API 未完成） | false |
| `DEBUG` | 启用详细日志 | false |
